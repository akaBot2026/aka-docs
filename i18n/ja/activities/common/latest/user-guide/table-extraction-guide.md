# テーブル抽出 — はじめに

> このドキュメントでは、**テーブル抽出**アクティビティを使用するための完全なワークフローを説明します。ウィザードの起動、テーブルの指定、列の設定、ページネーションのセットアップ、およびプロパティパネルでの詳細パラメーターの設定について説明します。

---

## 1. 概要

**テーブル抽出**は、アプリケーションのUI（Webブラウザ、Windowsデスクトップ、Javaなど）からテーブルデータを抽出し、結果を `DataTable` に格納するアクティビティです。

主な機能：

- 単一ページまたは**複数ページ**のテーブルの抽出（「次へ」/ページネーションボタン経由）。
- 各列のデータ型（`テキスト`、`数値`、`ブール値`）の設定。
- **行数**または**ページ数**による抽出の制限。

---

## 2. ワークフローへのアクティビティの追加

アクティビティを追加するには、次のワークフローに従います：
**akaBot Studio** > **ツールボックス**（検索: `テーブル抽出`）> ワークフローに**ドラッグ＆ドロップ** > **「テーブル抽出を開く」**をクリック。

![table-extraction.png](/static/img/table-extract-activity.png)

> [!IMPORTANT]
> **TargetElement** は必須の引数です。テーブルが指定されていない場合、アクティビティは実行時にバリデーションエラーを発生させます。

---

## 3. テーブル抽出ツール（ウィザード）を開く

設定ウィザード（テーブル抽出ツール）を開くには：

- アクティビティ本体の **「テーブル抽出を開く」** ボタンをクリックします。

ウィザードは、対象アプリケーションを遮らないよう**画面の上部コーナー**に位置する**テーブル抽出ウィザード**として開きます。この時点で：

![table-extraction-wizard.png](/static/img/table-extraction-wizard.png)

- Akabot Studio はタスクバーに**最小化**されます。
- 「指定」をクリックする前に、**対象アプリケーション**（ブラウザ、デスクトップソフトウェアなど）に**切り替える**必要があります。

---

## 4. ステップ1 — テーブル要素の指定

### 4.1. 「テーブル要素を指定」をクリック

ウィザードで **「テーブル要素を指定」** ボタンをクリックします。

- ウィザードが**最小化**されて画面が広がります。
- カーソルが**インスペクトモード**に入り、要素にホバーするとハイライト表示されます。

### 4.2. テーブル要素の選択

対象アプリケーションのテーブル要素にカーソルを移動してクリックします：

- **Web**: 表形式のデータを含むテーブルまたはテーブルのヘッダーをクリックします。
- **Windowsデスクトップ / Java**: グリッドまたはテーブルコントロールをクリックします。

![indicate-table-element.png](/static/img/indicate-table-element.png)

## 5. ステップ2 — 列の確認と設定

指定が成功すると、ウィザードにテーブルから検出された**列の一覧**が表示されます：

| フィールド           | 説明                                                                 |
| -------------------- | -------------------------------------------------------------------- |
| **IsSelected**       | 出力DataTableに列を含めるか除外するかのチェックボックス。             |
| **列名**             | テーブルヘッダー（先頭行）から読み取られた名前。変更可能。           |
| **設定（⚙）**        | テーブル列を設定するための設定ボタン（データ型、プレビューデータ）。  |

![table-extraction-wizard-indicate-element.png](/static/img/table-extraction-wizard-indicate-element.png)


### 5.1. 列の選択 / 選択解除

- 列のチェックボックスを**オフ**にする → 列は出力DataTableから**除外**されます。
- **オン**にして列を保持します。

### 5.2. DataType の設定

列の隣にある**設定（⚙）**アイコンをクリックすると、詳細設定パネルが開きます：

#### DataType = **Text**（デフォルト）

- 追加設定は不要です。
- 値は `string` として格納されます。

#### DataType = **Number**

| パラメーター             | デフォルト | 説明                                                          |
| ------------------------ | ---------- | ------------------------------------------------------------- |
| `Decimal Separator`      | `.`        | 小数点として使用する文字（例: `.` または `,`）。              |
| `Thousands Separator`    | `,`        | 千の位区切りとして使用する文字（例: `,` または `.`）。        |
| `Allow Null`             | `false`    | 有効にすると、解析に失敗した値は `null` として格納されます。  |
| `Default Value`          | `""`       | 解析失敗時のフォールバック値（Allow Null がオフの場合）。     |

#### DataType = **Boolean**

| パラメーター    | デフォルト           | 説明                                                    |
| --------------- | -------------------- | ------------------------------------------------------- |
| `True Values`   | `true,yes,1,oke`     | `true` として解釈する値のカンマ区切りリスト。           |
| `False Values`  | `false,no,0`         | `false` として解釈する値のカンマ区切りリスト。          |
| `Allow Null`    | `false`              | どのエントリにも一致しない場合に null を許可します。     |
| `Default Value` | `""`                 | 解析失敗時のフォールバック値。                          |

**解析失敗時の動作：**

| Allow Null | Default Value | 結果                    |
| ---------- | ------------- | ----------------------- |
| false      | _（空）_      | 例外をスローします      |
| false      | `0`           | `0` を使用します        |
| true       | `null`        | `DBNull` として格納     |
| true       | `0`           | `0` を使用します        |

> [!TIP]
> ウィザードの **「プレビュー」** ボタンをクリックすると、確認前に抽出されたすべてのデータをテーブルとして表示できます。

## 6. ステップ3 — 「次へ」ボタンの設定（ページネーション）

**「次へ」ボタン**セクション（折りたたみ可能）では、「次のページ」ボタンを設定します。これにより、アクティビティが複数ページのテーブルを自動的にページングし、すべての結果をマージできます。

### 6.1. 「次のリンクを指定」をクリック

ウィザードの **「次のリンクを指定」** をクリックします。

- ウィザードが最小化されます。対象アプリケーションのページネーションボタン（例: `>`、`次へ`、`»`）をクリックします。
- システムが以下を生成します：
  - ボタンの**厳密なセレクター**。
  - ボタンの**ファジーセレクター**。
  - **画像セレクター**（base64で格納されたボタンのスクリーンショット）。

![indicate-next-button.png](/static/img/indicate-next-button.png)

各セレクタータイプには、**有効/無効**を切り替えるチェックボックスと**精度スライダー**（0.0 → 1.0、デフォルト `0.5`）があります。

### 6.2. ページネーションの停止条件

以下のいずれかの条件が満たされると、アクティビティは自動的にページングを停止します：

- 「次へ」ボタンが**見つからない**場合（`ElementNotFoundException`）→ 正常停止。
- 「次へ」ボタンが**無効状態**の場合（`disabled` 属性、クラスに `disabled` を含む、`aria-disabled=true`、または `aastate=unavailable`）。
- 新しいページのデータが前のページと**同一**の場合（無限ループを防止）。
- 設定された**行数またはページ数の制限**に達した場合（以下のプロパティを参照）。

---

## 7. ウィザードの確認と終了

設定が完了したら：

- **確認**（`✓`）：すべての設定（セレクター、列設定、抽出メタデータ）をアクティビティに保存します。ウィザードが閉じます。
- **キャンセル**（`✗`）：すべての変更を破棄してウィザードを終了します。既存の設定は変更されません。

> [!WARNING]
> **キャンセル**をクリックすると、ウィザードを開いてから行った**すべての変更が破棄**されます。必ず**確認**をクリックして保存してください。

---

## 8. プロパティパネルリファレンス

ウィザードを閉じた後、アクティビティの**プロパティ**パネルを開いて追加の実行時パラメーターを設定します。

### 8.1. 入力 — 必須

| プロパティ       | 型       | 説明                                                                         |
| ---------------- | -------- | ---------------------------------------------------------------------------- |
| `TargetElement`  | `Target` | **[必須]** テーブル要素のセレクター。ウィザードによって自動入力されます。    |

### 8.2. 入力 — 任意

| プロパティ            | 型                    | デフォルト                     | 説明                                                                                   |
| --------------------- | --------------------- | ------------------------------ | -------------------------------------------------------------------------------------- |
| `NextButtonTarget`    | `Target`              | _（空）_                       | ページネーションボタンのセレクター。単一ページのテーブルの場合は空のままにします。      |
| `LimitExtractionTo`   | `ExtractionLimitType` | `NoLimit`                      | 抽出制限モード：`NoLimit`、`MaxRows`、または `MaxPages`。                              |
| `NumberOfItems`       | `Int32`               | `1000`（行）/ `200`（ページ） | 抽出する最大行数またはページ数。`LimitExtractionTo ≠ NoLimit` の場合に**必須**。       |
| `DelayBefore`         | `Int32`（ms）         | `200`                          | 実行開始前の待機時間（ms）。0以上でなければなりません。                                |
| `DelayAfter`          | `Int32`（ms）         | `300`                          | 実行完了後の待機時間（ms）。0以上でなければなりません。                                |
| `DelayBetweenPages`   | `Int32`（ms）         | `0`                            | ページネーション中のページクリック間の待機時間。0以上でなければなりません。             |
| `TimeoutMS`           | `Int32`（ms）         | `30000`                        | アクティビティ全体のタイムアウト。**3000ms以上でなければなりません**。                 |
| `ContinueOnError`     | `Boolean`             | `false`                        | `true` の場合、エラーはサイレントにログ記録され、実行が続行されます。                  |

### 8.3. 入力 / 出力

| プロパティ              | 型          | 説明                                                                                                             |
| ----------------------- | ----------- | ---------------------------------------------------------------------------------------------------------------- |
| `DestinationDataTable`  | `DataTable` | 結果を受け取るDataTable。既存のDataTableが渡された場合、新しい行はそこに**マージ**されます。                      |

### 8.4. 入力 — 自動生成

| プロパティ         | 説明                                                                                                                       |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| `ExtractMetadata`  | 抽出構造（列名、属性、CSSセレクターなど）を記述するXMLメタデータ。ウィザードによって自動生成されます。                    |
| `TableSettings`    | 列の型、データ型、名前、選択状態を含むXMLテーブル設定。ウィザードによって自動生成されます。                               |

> [!CAUTION]
> 内部XML構造を完全に理解していない限り、`ExtractMetadata` または `TableSettings` を**手動で編集しないでください**。誤った変更は抽出ロジックを破壊する可能性があります。

## 9. デフォルト値とバリデーションルール

### バリデーションエラー（ビルド / 実行時）

| 条件                                                               | エラー                                                                                         |
| ------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------- |
| `TargetElement` が設定されていない                                 | 必須バリデーションエラー                                                                       |
| `LimitExtractionTo ≠ NoLimit` だが `NumberOfItems` がnull         | 「抽出制限タイプが \{type\} に設定されている場合は、アイテム数を指定してください」             |
| `NumberOfItems ≤ 0`                                                | 「値は0より大きくなければなりません」                                                          |
| `TimeoutMS < 3000`                                                 | 「TimeoutMS は 3000 以上でなければなりません」                                                 |
| `DelayBefore < 0`                                                  | 「DelayBefore は 0 以上でなければなりません」                                                  |
| `DelayAfter < 0`                                                   | 「DelayAfter は 0 以上でなければなりません」                                                   |
| `DelayBetweenPages < 0`                                            | 「DelayBetweenPages は 0 以上でなければなりません」                                            |

### 内部デフォルト値（未設定の場合）

| プロパティ            | 内部デフォルト                               |
| --------------------- | -------------------------------------------- |
| `DelayBefore`         | `200 ms`                                     |
| `DelayAfter`          | `300 ms`                                     |
| `DelayBetweenPages`   | `0 ms`                                       |
| `TimeoutMS`           | `30000 ms`（`Constants.TimeoutMS`）          |
| `LimitExtractionTo`   | `NoLimit`                                    |
| `NumberOfItems`       | `1000`（MaxRowsモード）/ `200`（MaxPagesモード） |

### デザイン時とランタイムのテーブル構造の不一致

#### 問題

テーブル抽出アクティビティは、初期設定時に生成される基盤となるXMLメタデータに依存しています。対象アプリケーションのレイアウトが変わった場合（列の順序変更、名前変更、新しい列の挿入など）、事前設定されたセレクターとメタデータはライブDOMの構造と一致しなくなります。

簡単に言うと、**アクティビティが設定された時点（デザイン時）**と**自動化が実際に実行される時点（ランタイム）**の間でテーブルの構造が変わった場合、不一致が発生します。

**想定される構造（デザイン時）：**
![table-extract-design-time.png](/static/img/table-extract-design-time.png)

**実際の構造（ランタイム）：**
![table-extraction-runtime.png](/static/img/table-extraction-runtime.png)

このずれにより、抽出マッピングが壊れ、DataTableが不完全になったり、実行時例外が発生したりします。

#### 修正 — `TableSettings` から列定義をクリアする

最も安全な回避策は、`TableSettings` XMLからすべての `<Column>` 子ノードを**削除する**ことです。これにより、アクティビティは古いデザイン時の定義との照合を試みることなく、実行時にライブテーブルが提供するどの列構造でも受け入れるようになります。

**手順：**

1. アクティビティの**プロパティ**パネルで `TableSettings` プロパティを見つけます。
2. 式エディターボタン（`...`）をクリックして生のXML値を開きます。
3. `<Table ...>` ルート要素を見つけます。以下のような形になります：

```xml
<Table>
  <Column Name='id' ReferenceIdx='0' DataType='Number' DecimalSeparator='.' ThousandsSeparator=',' AllowNull='false' DefaultValue='' />
  <Column Name='First Name' ReferenceIdx='1' DataType='Text' />
  <Column Name='Last Name' ReferenceIdx='2' DataType='Text' />
  <Column Name='Product Name' ReferenceIdx='3' DataType='Text' />
  <Column Name='Quantity' ReferenceIdx='4' DataType='Number' DecimalSeparator='.' ThousandsSeparator=',' AllowNull='false' DefaultValue='' />
  <Column Name='Price' ReferenceIdx='5' DataType='Number' DecimalSeparator='.' ThousandsSeparator=',' AllowNull='false' DefaultValue='' />
  <Column Name='Total' ReferenceIdx='6' DataType='Number' DecimalSeparator='.' ThousandsSeparator=',' AllowNull='false' DefaultValue='' />
</Table>
```

4. すべての `<Column>...</Column>` 子ノードを**削除**して、`<Table>` ルート要素のみを残します：

```xml
<Table>
</Table>
```

5. 変更を保存して自動化を再実行します。
